## <a name="discussion"></a><span data-ttu-id="4e163-107">Tartışma</span><span class="sxs-lookup"><span data-stu-id="4e163-107">Discussion</span></span>  
 <span data-ttu-id="4e163-108">Uygulamalar, Kerberos, Özet veya HTTPS kullanarak NTLM kullanarak kimlik doğrulaması sırasında aktarım düzeyi güvenlik (TLS) kanalı ilk kurulur ve ardından kimlik doğrulama kullanarak güvenli kanal gerçekleşir.</span><span class="sxs-lookup"><span data-stu-id="4e163-108">When applications authenticate using Kerberos, Digest or NTLM using HTTPS, a Transport Level Security (TLS) channel is first established and then authentication takes place using the secure channel.</span></span> <span data-ttu-id="4e163-109">Ancak, SSL ile oluşturulan oturum anahtarı ile kimlik doğrulaması sırasında oluşturulan oturum anahtarı arasında hiçbir bağlama yok.</span><span class="sxs-lookup"><span data-stu-id="4e163-109">However, there is no binding between the session key generated by SSL and the session key generated during authentication.</span></span> <span data-ttu-id="4e163-110">Tüm MITM kendi istemci ve sunucu arasında kurmak ve sunucu güvenli kanal istemciden kurulan olmadığını olduğunu bilmesinin imkanı olduğundan taşıma kanalının güvenli olsa bile, istemciden istekleri iletme Başlat veya bir MITM.</span><span class="sxs-lookup"><span data-stu-id="4e163-110">Any MITM can establish itself between the client and the server and start forwarding the requests from the client, even when the transport channel itself is secure, because the server has no way of knowing whether the secure channel has been established from the client or a MITM.</span></span> <span data-ttu-id="4e163-111">Bu senaryoda sunucunun bir MITM olup olmadığını Algıla şekilde iç kimlik doğrulaması kanal ile dış TLS kanalına bağlamak için çözümüdür.</span><span class="sxs-lookup"><span data-stu-id="4e163-111">The solution in this scenario is to bind the outer TLS channel with the inner authentication channel such that the server can detect if there is a MITM.</span></span>
