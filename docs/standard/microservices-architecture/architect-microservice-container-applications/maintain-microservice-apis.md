# <a name="creating-evolving-and-versioning-microservice-apis-and-contracts"></a><span data-ttu-id="36628-103">Mikro hizmet API’leri ve anlaşmaları oluşturma, geliştirme ve sürüm oluşturma</span><span class="sxs-lookup"><span data-stu-id="36628-103">Creating, evolving, and versioning microservice APIs and contracts</span></span>

<span data-ttu-id="36628-104">Mikro hizmet API, hizmet ve istemcileri arasında bir sözleşmedir.</span><span class="sxs-lookup"><span data-stu-id="36628-104">A microservice API is a contract between the service and its clients.</span></span> <span data-ttu-id="36628-105">Bir mikro hizmet bağımsız olarak yalnızca çok önemli sözleşme, bu yüzden API sözleşmesinin bölmediğinizden varsa gelişmek mümkün olacaktır.</span><span class="sxs-lookup"><span data-stu-id="36628-105">You'll be able to evolve a microservice independently only if you do not break its API contract, which is why the contract is so important.</span></span> <span data-ttu-id="36628-106">Sözleşme değiştirirseniz, istemci uygulamalarınız veya API ağ geçidinizin etkiler.</span><span class="sxs-lookup"><span data-stu-id="36628-106">If you change the contract, it will impact your client applications or your API Gateway.</span></span>

<span data-ttu-id="36628-107">API tanımı doğasını hangi protokolünü kullandığınız bağlıdır.</span><span class="sxs-lookup"><span data-stu-id="36628-107">The nature of the API definition depends on which protocol you're using.</span></span> <span data-ttu-id="36628-108">Örneğin, Mesajlaşma kullanıyorsanız (gibi [AMQP](https://www.amqp.org/)), ileti türlerini API oluşur.</span><span class="sxs-lookup"><span data-stu-id="36628-108">For instance, if you're using messaging (like [AMQP](https://www.amqp.org/)), the API consists of the message types.</span></span> <span data-ttu-id="36628-109">HTTP ve RESTful hizmetlerini kullanıyorsanız, API URL ve istek ve yanıt JSON biçimleri oluşur.</span><span class="sxs-lookup"><span data-stu-id="36628-109">If you're using HTTP and RESTful services, the API consists of the URLs and the request and response JSON formats.</span></span>

<span data-ttu-id="36628-110">Ancak, ilk sözleşmeniz hakkında özenli olsa bile, bir hizmeti API zamanla değiştirmeniz gerekir.</span><span class="sxs-lookup"><span data-stu-id="36628-110">However, even if you're thoughtful about your initial contract, a service API will need to change over time.</span></span> <span data-ttu-id="36628-111">Bu durumda — ve özellikle API'NİZİN birden çok istemci uygulamalar tarafından kullanılan ortak bir API olduğunda — genellikle tüm istemcilerin yeni API sözleşmesine yükseltme tutamaz.</span><span class="sxs-lookup"><span data-stu-id="36628-111">When that happens—and especially if your API is a public API consumed by multiple client applications — you typically can't force all clients to upgrade to your new API contract.</span></span> <span data-ttu-id="36628-112">Genellikle yeni bir hizmet sözleşmesini eski ve yeni sürümleri aynı anda çalışan bir şekilde bir hizmet sürümlerini artımlı olarak dağıtmanız gerekebilir.</span><span class="sxs-lookup"><span data-stu-id="36628-112">You usually need to incrementally deploy new versions of a service in a way that both old and new versions of a service contract are running simultaneously.</span></span> <span data-ttu-id="36628-113">Bu nedenle, hizmet sürümü oluşturma için bir stratejiniz olması önemlidir.</span><span class="sxs-lookup"><span data-stu-id="36628-113">Therefore, it's important to have a strategy for your service versioning.</span></span>

<span data-ttu-id="36628-114">API değişiklikleri API'nize öznitelikleri veya parametreleri eklerseniz gibi küçük olduğunda, daha eski bir API kullanan istemciler geçin ve hizmetin yeni sürümle çalışır.</span><span class="sxs-lookup"><span data-stu-id="36628-114">When the API changes are small, like if you add attributes or parameters to your API, clients that use an older API should switch and work with the new version of the service.</span></span> <span data-ttu-id="36628-115">Gerekli olan eksik öznitelikler için varsayılan değerler sağlamak mümkün olabilir ve istemcilerin herhangi bir ek yanıt özniteliği yok saymak mümkün olabilir.</span><span class="sxs-lookup"><span data-stu-id="36628-115">You might be able to provide default values for any missing attributes that are required, and the clients might be able to ignore any extra response attributes.</span></span>

<span data-ttu-id="36628-116">Ancak, bazen büyük ve uyumsuz değişiklik, bir hizmeti API'sine gerekir.</span><span class="sxs-lookup"><span data-stu-id="36628-116">However, sometimes you need to make major and incompatible changes to a service API.</span></span> <span data-ttu-id="36628-117">İstemci uygulamaları veya hizmetleri hemen yeni sürüme yükseltmek için zorlamak mümkün olmayabilir çünkü bir hizmeti API eski sürümleri için belirli bir süre desteklemesi gerekir.</span><span class="sxs-lookup"><span data-stu-id="36628-117">Because you might not be able to force client applications or services to upgrade immediately to the new version, a service must support older versions of the API for some period.</span></span> <span data-ttu-id="36628-118">Bir HTTP tabanlı mekanizması gibi REST kullanıyorsanız, bir URL veya bir HTTP üst bilgisi API sürüm numarasını ekleyin yaklaşımdır.</span><span class="sxs-lookup"><span data-stu-id="36628-118">If you're using an HTTP-based mechanism such as REST, one approach is to embed the API version number in the URL or into an HTTP header.</span></span> <span data-ttu-id="36628-119">Sonra iki sürümü de aynı hizmet örneği hizmetinde aynı anda uygulama veya her bir API sürümünü işlemek farklı örnekleri dağıtma arasında karar verebilirsiniz.</span><span class="sxs-lookup"><span data-stu-id="36628-119">Then you can decide between implementing both versions of the service simultaneously within the same service instance, or deploying different instances that each handle a version of the API.</span></span> <span data-ttu-id="36628-120">Bunun iyi bir yaklaşımdır [Dünyası deseni](https://en.wikipedia.org/wiki/Mediator_pattern) (örneğin, [MediatR Kitaplığı](https://github.com/jbogard/MediatR)) farklı sürümler bağımsız işleyicileri ayırmak için.</span><span class="sxs-lookup"><span data-stu-id="36628-120">A good approach for this is the [Mediator pattern](https://en.wikipedia.org/wiki/Mediator_pattern) (for example, [MediatR library](https://github.com/jbogard/MediatR)) to decouple the different implementation versions into independent handlers.</span></span>

<span data-ttu-id="36628-121">Son olarak, bir REST mimarisi kullanıyorsanız [gösterimde](https://www.infoq.com/articles/mark-baker-hypermedia) hizmetlerinizi ve evolvable API'leri sağlayan sürüm oluşturma için en iyi bir çözümdür.</span><span class="sxs-lookup"><span data-stu-id="36628-121">Finally, if you're using a REST architecture, [Hypermedia](https://www.infoq.com/articles/mark-baker-hypermedia) is the best solution for versioning your services and allowing evolvable APIs.</span></span>
